# SOP-AUTO-20260606T025000 : Restore Failure - Backup File Not Found

## Classification
- **Category**: RESTORE
- **Severity**: P1

## Problem Description
A RESTORE operation for the database `salesdb` on SQL Server 2025 (Windows VM, 10.10.0.5) failed because the backup file could not be found at the specified URL path. The error messages included:
- "Cannot open backup device"
- "The system cannot find the file specified"

The restore uses URL-based restore from Azure Blob Storage container `backupfromazsqlmi` in storage account `2276259blob`, leveraging a SAS credential for access. The operation requires a valid `.bak` file less than 48 hours old. The automated tool `restore_database` failed at locating the exact valid backup file path for the COPY_ONLY backup of `salesdb`.

## Root Cause Analysis
Possible root causes include:

1. **Incorrect backup file path or filename syntax**:
   - Backup files might have a different naming convention than expected by the restore job.
   - The tool’s discovered backup filename could be stale or incorrect.

2. **Backup file missing or outdated**:
   - No backup file for `salesdb` within the last 48 hours exists in the blob container.
   - The COPY_ONLY backup might not have been pushed or completed successfully from Azure SQL Managed Instance.

3. **Blob container or storage account access issues**:
   - The SAS token used for RESTORE FROM URL might be expired or have insufficient permissions.
   - Network rules or firewall settings on the storage account preventing access from the SQL Server VM.

4. **Blob listing failure in automation**:
   - The tool running on the Linux dev VM may have failed to accurately list or parse the blob storage container contents.
   - Possible transient network errors or permissions problems when enumerating blobs.

5. **Restore URL construction error**:
   - The restore command URL parameter for the backup file might have incorrect syntax or contain URL-encoding errors.

6. **Mismatch in source and target environments**:
   - Backup files generated by Azure SQL MI might have internal filename patterns differing from local SSMS naming expectations.
   - Restore routine assumes a certain naming scheme that doesn't match actual blob names.

## Resolution Steps

1. **Verify blob storage container contents**:
   - Connect to `2276259blob` storage account using Azure CLI or Azure Portal.
   - List blobs in container `backupfromazsqlmi` and identify `.bak` files related to `salesdb`.
     ```bash
     az storage blob list --account-name 2276259blob --container-name backupfromazsqlmi --query "[?contains(name, 'salesdb')]" --output table
     ```
   - Confirm presence of a COPY_ONLY `.bak` file not older than 48 hours.

2. **Validate backup file name and full Blob URL**:
   - Note the exact blob name and construct a URL in the form:
     ```
     https://2276259blob.blob.core.windows.net/backupfromazsqlmi/<backup_file_name>.bak
     ```
   - Ensure no typos, extra path segments, or case sensitivity issues.

3. **Confirm SAS token validity**:
   - Check that the SAS token used in RESTORE FROM URL hasn't expired and has at least Read and List permissions.
   - If expired or insufficient, generate a new SAS token via Azure Portal or CLI:
     ```bash
     az storage blob generate-sas --account-name 2276259blob --container-name backupfromazsqlmi --name <backup_file_name>.bak --permissions r --expiry <future-date-in-ISO-format> --https-only
     ```
   - Test SAS URL accessibility using a browser or curl from the SQL Server VM or dev Linux VM:
     ```bash
     curl -I "<SAS_URL>"
     ```
   - Confirm HTTP 200 OK response.

4. **Verify network connectivity and firewall rules**:
   - From SQL Server VM (10.10.0.5), ensure outbound network connectivity to Azure Blob endpoints is allowed.
   - Confirm no NSG or firewall blocks to `*.blob.core.windows.net`.

5. **Test RESTORE command manually**:
   - On SQL Server VM, using SSMS or sqlcmd, run a minimal RESTORE command using the verified SAS URL:
     ```sql
     RESTORE DATABASE salesdb FROM URL = 'https://2276259blob.blob.core.windows.net/backupfromazsqlmi/<backup_file_name>.bak?<SAS_token>' WITH FILE = 1, NORECOVERY;
     ```
   - Confirm that the backup file opens successfully and no "file not found" errors occur.

6. **Update automation tool parameters**:
   - Modify the `restore_database` tool input to reference the confirmed exact blob filename and URL.
   - Ensure the tool refreshes or re-lists blob container contents each run to avoid using cached or stale filenames.

7. **Document and communicate**:
   - Document the confirmed correct backup file path and SAS URL for future restores.
   - Notify the team about any changes related to backup naming or SAS token management.

## Prevention

- Implement automatic validation of backup presence and SAS token freshness before restore attempts.
- Schedule regular backup file verification jobs running on the dev VM to alert if backups are missing from blob storage.
- Use automated blob listing in restore tools that handle all edge cases of file naming and expiration.
- Monitor network/firewall rules to prevent connectivity issues between the SQL Server VM and Azure Blob Storage service.
- Ensure backup jobs on Azure SQL Managed Instance push COPY_ONLY backups consistently and naming conventions stay documented.
- Automate SAS token renewal and distribution to all systems performing restore operations.

## Suggested Tool Improvement

- Enhance `restore_database` automation to:
  - Dynamically list and validate all candidate backup files from Azure Blob Storage before restore.
  - Automatically select the most recent valid COPY_ONLY backup within defined retention.
  - Validate SAS token expiry and request regeneration if expired.
  - Include detailed logging of blob storage listing and RESTORE command attempts.
  - Support retry logic for transient connectivity or permission errors.
  - Provide an explicit dry-run mode to verify file paths and access without initiating restore.
