# SOP-AUTO-2024-06-09-01 : RESTORE Failure - Missing Backup File Path

## Classification
- **Category**: RESTORE
- **Severity**: P1
- **Auto-Generated**: Yes

## Problem Description
The restore operation for database `salesdb` on SQL Server 2025 (Windows VM 10.10.0.5) failed because the specified backup file path `D:\backups\salesdb_copy_only.bak` was not found or inaccessible. The system was unable to locate the backup file, resulting in a failed restore.

## Root Cause Analysis
Considering the Azure cloud environment and architecture:

- **Backup Source**: Backups are generated from Azure SQL Managed Instance (SQL MI) as `.bak` files.
- **Backup Storage**: Backup files are pushed **directly to Azure Blob Storage** (`backupfromazsqlmi` container in storage account `2276259blob`).
- **Restore Method**: The restore should occur using `RESTORE FROM URL` with SAS token authentication to access the blob storage container, **not** from the local disk.
- **Failure Cause**: The error references a local disk path (`D:\backups\salesdb_copy_only.bak`). This indicates:
  - The restore command is incorrectly targeting a local disk path instead of restoring from Blob Storage.
  - Possible misconfiguration of the restore command or automation script.
- Other possible contributing issues:
  - Blob storage path or file naming mismatch (`salesdb_copy_only.bak` vs actual blob name).
  - SAS token may be expired or not supplied in restore command causing fallback or failure.
  - The automation tool might be failing to list blob container files, leading to incorrect local path fallback.
  - Network/firewall restrictions preventing access to blob URLs from SQL Server VM.
  - The SQL Server service account does not have proper read permission if local disk usage was intended.
  
Given that the architecture mandates using RESTORE FROM URL via SAS credentials and no local disk copy of the `.bak` files is confirmed, the root cause is primarily **the restore command referencing a local non-existent backup file path instead of the blob storage URL**.

## Resolution Steps

1. **Verify Backup File Existence in Blob Storage**
   - Use Azure CLI or Azure Storage Explorer to confirm the exact filename of the latest `.bak` file in `backupfromazsqlmi` container.
   - Confirm it is a valid recent COPY_ONLY backup for `salesdb` (preferably generated within last 48 hours).
   ```bash
   az storage blob list --account-name 2276259blob --container-name backupfromazsqlmi --query "[].name"
   ```
   
2. **Validate SAS Token Validity and Permissions**
   - Confirm SAS token used for the restore has read (`r`) and list (`l`) permissions.
   - Ensure SAS token is not expired.

3. **Update/Correct the Restore Command**
   - Modify the restore command to use `RESTORE FROM URL = 'https://2276259blob.blob.core.windows.net/backupfromazsqlmi/salesdb_copy_only.bak' WITH CREDENTIAL = <SAS_Credential>;`
   - Avoid referencing local paths like `D:\backups\...` unless local copies exist and are accessible.
   
4. **Check Network & Firewall Access**
   - Verify SQL Server VM at 10.10.0.5 can access Azure Blob URLs.
   - Ensure no firewall or NSG rules block outbound HTTPS traffic to blob storage endpoints.

5. **Ensure Automation Workflow Correctly Lists Blobs**
   - Validate the Linux dev VM automation script uses correct Azure SDK or CLI to list blobs.
   - Cross-check backup filename discovery logic to prevent fallback on local paths.

6. **If Needed, Download Backup to Local Disk (Temporary)**
   - As a last resort, download the `.bak` file from blob storage to the VM’s `D:\backups\` folder.
   - Grant SQL Server service account read permissions on the folder.
   - Adjust restore command to point to the local `.bak` file path.
   - Note: This step is non-ideal and should be temporary.

7. **Retry Restore Operation**

## Prevention
- Enforce standardization to **always restore from URL using Azure Blob Storage with SAS** — do **not** use local disk paths.
- Implement monitoring jobs to verify:
  - Existence of backup files in blob storage container.
  - SAS token validity and permissions.
  - Network connectivity between SQL Server VM and Azure Blob.
- Document backup file naming conventions strictly aligned with source DB names.
- Automate alerting on any backup file creation failures or naming mismatches.
- Version control and review all restore automation scripts to prevent hard-coded local paths.
- Periodically test restores in non-production environments to validate pipeline integrity.

## Suggested Tool Improvement
- Enhance the backup restore automation tool to:
  - Only accept and validate blob storage URLs as backup sources.
  - Automatically verify blob container listing succeeds before generating restore commands.
  - Automatically refresh or validate SAS tokens before restore.
  - Generate detailed errors indicating SAS and network issues rather than fallback errors referencing local paths.
  - Include configuration parameter validation to prevent misconfigured local file paths usage.
  - Provide a dry-run listing feature to confirm backups available in blob prior to restore attempts.
