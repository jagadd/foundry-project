# SOP-AUTO-20260609T114349 : Restore Failure Due to Disk Space Check Service Unavailability

## Classification
- **Category**: RESTORE
- **Severity**: P2
- **Auto-Generated**: Yes

## Problem Description
Attempt to restore database `EnterpriseSales` from Azure SQL Managed Instance backup stored in Azure Blob Storage to an on-premises SQL Server 2025 instance running on a Windows VM (10.10.0.5) failed. The restore process was blocked because the automated pre-flight disk space check service on the target VM was unavailable and could not verify sufficient disk space for the restore operation. The backup file `.bak` was found successfully in Blob storage, but the disk check step interruption prevented the restore from proceeding.

## Root Cause Analysis
- The backup file for `EnterpriseSales` exists in the Azure Blob container `backupfromazsqlmi` on storage account `2276259blob` and is accessible; backups are properly pushed as `.bak` files.
- The restore method uses `RESTORE FROM URL` with a SAS credential, confirming connectivity and permission to the backup file from the SQL Server VM, as no errors about SAS token or network/firewall were reported.
- The automation, running on a Linux dev VM via pyodbc, successfully enumerated the backup files in the container, so blob listing and SAS token validity for read/list operations are intact.
- The failure occurred at the disk space check step on the target Windows VM (10.10.0.5). The disk space check service, which validates if sufficient free space exists to restore the database files, reported unavailability (`SIMULATED: Disk check service unavailable`).
- Without the disk space check service running or reachable, the automation safely refuses to proceed with restore, preventing possible data loss or restore failure due to insufficient disk space.
- Potential causes for this service unavailability include:
  - The disk space check service on the Windows VM is stopped, crashed, or not installed.
  - Network/firewall rules block communication between the automation host or controller and the disk space check service on the Windows VM.
  - Service account permissions preventing access to disk usage info.
- No indication of issues with backup presence, naming conventions, or SAS token expiry.

## Resolution Steps
1. **Verify Disk Space Check Service Status on Target VM**
   - Connect remotely to Windows VM (10.10.0.5).
   - Confirm the disk space check service/process is running and responsive.
   - If stopped or crashed, restart the service.
2. **Verify Network Connectivity**
   - From the Linux dev VM or automation host, validate network connectivity and firewall rules allow access to the disk space check service port on the Windows VM.
   - Adjust firewall settings if necessary.
3. **Verify Permissions**
   - Ensure the disk space check service runs under an account with permission to query disk space on the VM.
4. **Test Disk Space Check Manually**
   - Run disk space queries manually on 10.10.0.5 to confirm service functionality.
5. **Re-run the Restore Automation**
   - Once the disk space check service is confirmed operational, repeat the restore operation.
6. **Implement Monitoring and Alerts**
   - Set up automated monitoring for the disk space check service health and alert on unavailability to avoid future manual triage delays.

## Prevention
- Implement health checks and an automated alert system for the disk space check service on target VMs.
- Periodically verify connectivity and permission levels needed for disk space validation.
- Document recovery steps for disk space check service failure and automate failover or self-healing scripts.
- Coordinate change management for network/firewall configurations to avoid unintentionally blocking service communication.

## Suggested Tool Improvement
- Enhance the restore automation to:
  - Retry disk space check service calls with exponential backoff before aborting.
  - Fall back to an alternate lightweight disk space check method if the primary service is unavailable.
  - Provide more granular diagnostics and remediation suggestions in error messages.
  - Include a “force restore” flag allowing overrides after operator confirmation with documented risk.
