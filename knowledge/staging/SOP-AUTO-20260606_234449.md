# SOP-AUTO-20260606T234413 : Restore Operation Failed - Backup File Path Not Provided

- **Category**: RESTORE

## Problem Description
A RESTORE operation for the database **EnterpriseSales** on the target SQL Server 2025 VM failed. The error indicated the absence of a specific backup file (.bak) path, with the operation being rejected by the system operator due to the missing file reference. The restoration process expected a backup file specified by URL with SAS credentials from Azure Blob Storage, but the file was not provided or not found.

## Root Cause Analysis
- The restore method relies exclusively on **RESTORE FROM URL** using SAS credentials to Azure Blob Storage, without local disk involvement.
- Possible causes include:
  - **Backup file missing or incorrectly named** in the Azure Blob Storage container (`backupfromazsqlmi` in storage account `2276259blob`). The automation tool may not have correctly discovered or listed the backup file for `EnterpriseSales`.
  - **Incorrect or expired SAS token** used by the restore operation, causing failure to access or list blobs inside the container.
  - **Network or firewall restrictions** between the Linux dev VM (automation host) or the SQL Server VM and the Azure Blob Storage endpoint, preventing proper backup listing or file access.
  - **Mismatch in backup file naming conventions**: The tool may expect a backup filename matching the source DB name `EnterpriseSales` but the actual blob name diverges (e.g. with timestamps, prefixes, or suffixes).
  - **Blob listing failure or incomplete auto-discovery**: The automation script that enumerates backup files in the container might have failed or missed the correct backup file.
  - The system operator’s rejection may also indicate a procedural or permissions issue on the SQL Server side preventing restore without explicit file path specification.

## Resolution Steps
1. **Verify Backup File Presence and Naming**  
   - Log in to Azure Portal or use Azure CLI/PowerShell to list blobs in the container `backupfromazsqlmi` of storage account `2276259blob`:  
     ```bash
     az storage blob list --container-name backupfromazsqlmi --account-name 2276259blob --output table
     ```  
   - Confirm the presence of a recent full backup file for `EnterpriseSales` (.bak file) and note its exact blob name.

2. **Check SAS Token Validity and Permissions**  
   - Verify that the SAS token used for RESTORE FROM URL is valid, unexpired, and has at least Read (`r`) and List (`l`) permissions for the blob container and backup file.
   - Regenerate SAS token if expired or insufficient permissions exist.

3. **Validate Network Connectivity**  
   - From the SQL Server 2025 VM (10.10.0.5), verify outbound connectivity to Azure Blob Storage endpoint of `2276259blob` account (e.g. using `Test-NetConnection` PowerShell or equivalent).  
   - Ensure there are no firewall or NSG rules blocking HTTPS port 443 to blob endpoints.

4. **Explicitly Specify Backup File Path in Restore Command**  
   - Modify the restore operation to include the full backup file URL with the SAS token, for example:  
     ```sql
     RESTORE DATABASE [EnterpriseSales] FROM URL = 'https://2276259blob.blob.core.windows.net/backupfromazsqlmi/EnterpriseSales_20260605.bak?sv=...' WITH ...
     ```  
   - This bypasses ambiguity and satisfies system operator requirements.

5. **Test Blob Listing via Automation Tool**  
   - On the Linux dev VM, confirm that the automation script correctly lists blobs from the container using the Azure Storage SDK or CLI.
   - Check for any errors or exceptions related to blob enumeration.

6. **Retry Restore Operation**  
   - After completing above checks and corrections, rerun the restore process specifying the validated full backup URL.

## Prevention
- Implement monitoring/alerting on backup uploads to Blob Storage to ensure backup files are correctly named and uploaded.
- Automate SAS token renewal with adequate permissions and keep token expiration in sync with restore window.
- Enhance network validation scripts as part of the automation to verify connectivity before restore attempts.
- Standardize and document naming conventions for backup files and enforce them in the backup job on Azure SQL Managed Instance.
- Ensure restore scripts always explicitly specify full backup file URLs, never rely on implicit discovery alone.
- Periodically test automation tool blob listing and restore paths in staging environments.

## Suggested Tool Improvement
- Enhance the `restore_database` tool to:
  - Automatically fail early with descriptive errors if no matching backup file is found during blob listing.
  - Support automatic discovery and prompt for confirmation of correct backup file URL prior to executing RESTORE.
  - Include SAS token validation and refresh feature to prevent using expired tokens.
  - Incorporate network connectivity tests and retry logic.
  - Log full URLs and commands used for transparency and troubleshooting.
