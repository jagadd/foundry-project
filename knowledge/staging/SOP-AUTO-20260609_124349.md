## Problem Description
The restore operation for database **EnterpriseSales** on the target SQL Server 2025 VM failed with the error message:  
`Exclusive access could not be obtained because the database is in use by active connections (3 sessions).`  

Despite the pre-flight checks indicating zero active connections and sufficient disk space, the final restore failed due to three active sessions connected to the database blocking the exclusive lock needed for the RESTORE operation.

## Root Cause Analysis
- The pre-flight tooling verified zero active connections on the target database **EnterpriseSales** immediately before initiating restore.
- However, during the restore attempt, three active sessions maintained or re-established connections:
  - SPIDs 55 & 58: login `app_user`, program `AppPool_v2`
  - SPID 61: login `report_svc`, program `ReportService`
- These sessions prevented exclusive access causing SQL Server to abort the RESTORE command.
- Possible contributing factors:
  - Timing gap between active connection check and restore execution – connections may have been initiated or persisted during this gap.
  - Automated jobs, monitoring tools, or application pools that automatically connect to the database without being suspended prior to restore.
  - No pre-restore mechanism was in place to forcibly disconnect or kill active connections.
- The backup file **EnterpriseSales_20260608_1021.bak** was present and accessible on Azure Blob Storage (container: `backupfromazsqlmi`, storage account: `2276259blob`) using correct SAS URL credentials.
- Disk space on the target VM was sufficient (41.5 GB free on D: drive).
- Network connectivity and blob access were validated and confirmed successful by blob listing and backup verification commands.
- Therefore, failure is clearly rooted in database access concurrency during restore rather than storage, network, or backup file issues.

## Resolution Steps
1. **Suspend incoming application traffic/connections:**
   - Notify application owners and schedule downtime to prevent new connections during the restore window.
   - Temporarily disable AppPool_v2 in IIS or stop dependent services that auto-connect (e.g., ReportService).

2. **Kill existing sessions blocking the restore:**
   - Connect to target SQL Server instance remotely.
   - Run the following query to identify active sessions for `EnterpriseSales`:

     ```sql
     SELECT session_id, login_name, program_name FROM sys.dm_exec_sessions 
     WHERE database_id = DB_ID('EnterpriseSales');
     ```

   - Kill all sessions:

     ```sql
     ALTER DATABASE EnterpriseSales SET SINGLE_USER WITH ROLLBACK IMMEDIATE;
     ```

3. **Perform the RESTORE using URL and SAS credentials again:**

   ```sql

   FROM URL = 'https://2276259blob.blob.core.windows.net/backupfromazsqlmi/EnterpriseSales_20260608_1021.bak'
   WITH CREDENTIAL = 'YourSASCredential',
   REPLACE;
   ```

4. **After successful restore, set the database back to MULTI_USER mode:**

   ```sql
   ALTER DATABASE EnterpriseSales SET MULTI_USER;
   ```

5. **Re-enable application services and monitor for new connections.**

6. **Verify the database state and data consistency after restore completes successfully.**

## Prevention
- Implement a pre-restore step in automation to forcibly disconnect all active connections and set the database to SINGLE_USER mode immediately before RESTORE.
- Coordinate restore windows with application teams to avoid race conditions from auto-reconnecting services.
- Use SQL Server Resource Governor or connection throttling if possible to temporarily block new connections.
- Enhance monitoring to alert if sessions persist in databases scheduled for restore attempts.
- Consider maintenance mode flags or sticky tickets that block app pools from launching during restores.

## Suggested Tool Improvement
- Augment restore automation (running from Linux VM via pyodbc) to:
  - Detect and forcibly kill active connections prior to RESTORE.
  - Automatically switch database to SINGLE_USER mode with ROLLBACK IMMEDIATE.
  - Retry restore upon transient exclusive access errors, with configurable retry count and delay.
  - Log detailed session information and notify operators of blocking sessions.
- Add a step to lock down application access (e.g., via API or infrastructure automation to pause app pools) integrated into the restore workflow to avoid human error.
- Implement time synchronization (close gap) between active connection check and restore execution to reduce race conditions.
