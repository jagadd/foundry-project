The automated restore operation for the database "EnterpriseSales" on the SQL Server 2025 target VM failed to proceed because the disk space check tool reported service unavailability. ## Root Cause Analysis
- The disk space check service, which is expected to run within the environment or as a remotely accessible service, was unavailable or unreachable at the time of the restore attempt, leading to a failure in the disk space validation step.
- Network or service issues on the Linux dev VM or the target Windows VM may have prevented this service from executing successfully.
- The disk space check failure prevented the automation from proceeding, since it cannot confirm there is enough space for the restore, increasing risk of restore failure or partial restore.
- Backup discovery and verification are successful, ruling out issues with Azure Blob Storage connectivity, SAS token validity, container naming, or backup file availability.
- The target database "EnterpriseSales" is online and idle, confirming the environment is ready for the restore operation except for the disk space verification.

## Resolution Steps
1. **Manual Disk Space Verification**
   - Log into the Windows VM (10.10.0.5) hosting SQL Server 2025.
   - Check available free space on the volume(s) expected to store the database files.
     ```powershell
     Get-PSDrive -PSProvider FileSystem
     ```
   - Confirm available disk space is greater than the backup size (~1.69 MB for latest backup as per blobs) plus overhead for restore operation.

2. **Check Disk Space Monitoring Service**
   - Identify the disk space check service or script used by the automation.
   - Investigate any errors or outages on the Linux dev VM or closely related services that might enforce disk checks.
   - Restart or fix the service if possible; verify logs for error details.

3. **Retry Disk Space Check Tool**
   - From the Linux dev VM, try running the disk space check tool/command manually.
   - Confirm if it is now responding.
   - If it works, re-run the automated restore.

4. **Bypass Disk Space Check with Caution**
   - If immediate restore is critical and manual disk space verification confirms enough capacity, consider overriding the disk space check temporarily.
   - Document this bypass step and notify stakeholders of the increased risk.

5. **Validate SAS Permissions and Network Connectivity (Optional)**
   - Though not failing here, verify the SAS token for Azure Blob Storage has not expired and has necessary permissions.
   - Confirm network connectivity intermittently between Linux VM and Azure Blob Storage and between Linux VM and Windows VM.

6. **Initiate Restore**

## Prevention
- Implement monitoring and alerting on the disk space check service's availability and responsiveness.
- Schedule periodic health checks for all automation prerequisite services including disk space checking.
- Automate alerts on disk space running low on target SQL server volumes.
- Maintain up-to-date SAS credentials with proper expiry management.
- Document and maintain clear service dependency maps for all automation tools.
- Consider fallback mechanisms in the automation to detect and report service unavailability more gracefully.

## Suggested Tool Improvement
- Enhance the disk space check tool to include retries with exponential backoff before giving failure status.
- Implement a fallback mechanism to query free disk space directly via ODBC or PowerShell commands remotely if the primary disk check service is unavailable.
- Add detailed logging and alerting for disk check failures to reduce manual triage time.
- Incorporate a “disk space check manual override” flag in the automation with approval workflows to expedite restores during disk check service outages.
- Extend blob listing and backup verification to also validate SAS token expiry time and alert proactively on nearing expiration.
