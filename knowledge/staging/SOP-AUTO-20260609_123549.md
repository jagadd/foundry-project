## Problem Description
A restore operation for the database `EnterpriseSales` from Azure SQL Managed Instance backup stored in Azure Blob Storage to SQL Server 2025 on a Windows VM (10.10.0.5) failed. The restore attempt could not obtain exclusive access to the target database because there are active connections preventing the restore process from starting.

```

Msg 3013, Level 16, State 1: RESTORE DATABASE is terminating abnormally.
Active connections count: 3 from sessions (SPIDs 55, 58, 61) with logins 'app_user' and 'report_svc'.
```

- No indications of issues with the backup retrieval via RESTORE FROM URL or SAS token validity.

## Resolution Steps
1. **Manually Identify and Kill Active Connections:**
   - Connect to the target SQL Server instance (10.10.0.5) with sufficient permissions.
   - Run the following T-SQL to identify active connections:
     ```sql
     SELECT session_id, login_name, program_name
     FROM sys.dm_exec_sessions
     WHERE database_id = DB_ID('EnterpriseSales');
     ```
   - Terminate all active sessions connected to `EnterpriseSales`:
     ```sql
     ALTER DATABASE EnterpriseSales SET SINGLE_USER WITH ROLLBACK IMMEDIATE;
     ```
     or alternatively:
     ```sql
     EXEC sp_who2;
     KILL <session_id>; -- for each active session found
     ```
2. **Perform Restore Using RESTORE FROM URL Again:**
   - Use your existing automation or manual restore command ensuring you specify the URL to the Azure Blob storage and SAS credential.
   - Example command outline:
     ```sql
     RESTORE DATABASE EnterpriseSales
     FROM URL = 'https://2276259blob.blob.core.windows.net/backupfromazsqlmi/EnterpriseSales_20260608_1021.bak'
     WITH REPLACE, STATS = 5;
     ```
3. **Set Database Back to MULTI_USER:**
   After successful restore, execute:
   ```sql
   ALTER DATABASE EnterpriseSales SET MULTI_USER;
   ```
4. **Verify the Restore Completion and Database Accessibility:**
   - Check database state is ONLINE.
   - Verify no blocking or access issues.
   - Confirm application connectivity.

## Prevention
- Prior to restore attempts, automate disconnection of active sessions:
  - Extend pre-flight checks in automation to query active connections and force disconnect if needed.
  - Use SQL commands to set the target database to SINGLE_USER mode prior to restore.
- Schedule restore operations during maintenance windows with minimized user activity.
- Inform application owners and reporting teams to pause their activities during restore windows.
- Implement retry logic in automation to handle transient active connections by waiting or forcing disconnects.
- Log and alert when active connections block restore.
