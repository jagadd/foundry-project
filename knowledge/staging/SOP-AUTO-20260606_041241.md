- **Category**: RESTORE

## Problem Description
The RESTORE operation for the database `sales34343` on the SQL Server 2025 target VM failed because the required `.bak` backup file named `sales34343.bak` was not found in the configured Azure Blob Storage container `backupfromazsqlmi` of storage account `2276259blob`. This prevented the restore process from proceeding as it relies exclusively on a valid backup file accessible via URL with SAS credentials.

## Root Cause Analysis
- **Backup File Missing or Misnamed**: The backup file `sales34343.bak` may not have been pushed from the Azure SQL Managed Instance (SQL MI) to the blob container or is named differently from the expected filename.
- **Incorrect Blob Container or Path**: The restore process expects the file in the container `backupfromazsqlmi` at the root or specified path. The file may reside in a subfolder or a different container.
- **SAS Token Issues**: The SAS token used to provide URL access could be expired, malformed, or not scoped to the correct container or file path, preventing visibility or access to the backup file.
- **Blob Listing or Connectivity Issues**: The automation tool running from the Linux Dev VM uses blob listing to discover backups. Any network, firewall, or permission issues could cause failure to list or verify the presence of the `.bak` file.
- **Naming Convention Mismatch**: There might be a mismatch between the SQL MI database name (`sales34343`) and the actual backup file naming convention used during the backup push. For example, additional timestamp or GUID suffixes might be appended to the filename.
  
## Resolution Steps
1. **Verify Backup File Existence and Naming in Blob Storage**
   - Use Azure Portal, Azure CLI (`az storage blob list`), or Azure Storage Explorer to inspect the `backupfromazsqlmi` container in the `2276259blob` storage account.
   - Confirm if `sales34343.bak` or similarly named files exist.
   - If multiple files with timestamps or suffixes exist (e.g., `sales34343_20260605.bak`), adjust the restore URL or tool parameters accordingly.
2. **Confirm Backup File Upload From SQL MI**
   - On Azure SQL Managed Instance, validate that the backup job/process has successfully generated and uploaded the `.bak` file.
   - Query backup history and check automation logs responsible for uploading backups.
   - Retry or initiate a fresh backup and upload if the backup file is missing.
3. **Check SAS Token Validity and Permissions**
   - Verify the SAS token used has not expired and provides read/list access permissions to `backupfromazsqlmi` container.
   - Regenerate the SAS token with appropriate permissions scoped to the correct container and file if necessary.
   - Test access to the file URL using tools like `curl` or `Invoke-WebRequest` to confirm accessibility.
4. **Review Restore Tool Configuration and Blob Path**
   - Confirm that the restore automation running on the Linux Dev VM uses the correct container and storage account names.
   - Validate the exact URL constructed in the RESTORE command, including any path prefixes to the `.bak` file.
   - Modify the restore script or parameters if the backup file resides in a subfolder within the container.
5. **Check Network and Firewall Settings**
   - Ensure the Linux Dev VM and SQL Server target VM have outbound connectivity to Azure Blob Storage endpoints.
   - Confirm no firewall or NSG rules block access to the storage account endpoint.
6. **Retry the Restore Operation**
   - After confirming and/or rectifying any issues above, retry the RESTORE operation using the updated blob URL and valid SAS token.

## Prevention
- Implement monitoring and alerting on backup upload jobs from SQL MI to Azure Blob storage, ensuring successful and timely creation of `.bak` files.
- Enforce a standardized backup file naming convention and update the restore automation to handle any dynamic naming (e.g., timestamps).
- Maintain scheduled SAS token renewal processes or use Azure AD authentication where possible for Blob Storage access.
- Automate periodic verification of backup file presence in Blob Storage container before attempting restores.
- Document and validate network/firewall rules to allow uninterrupted connectivity between all involved components.

## Suggested Tool Improvement
- Enhance the restore automation tool (`restore_database`) to:
  - Perform a dynamic blob container listing and accept alternative backup filenames or pattern matching.
  - Provide detailed diagnostics on SAS token permissions and expiry before performing the RESTORE.
  - Allow configurable blob path inputs or auto-detection of backup file locations within the container.
  - Implement preflight checks that verify both backup presence and accessibility over the network prior to initiating the restore.
