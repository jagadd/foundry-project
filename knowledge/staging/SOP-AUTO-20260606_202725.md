# SOP-AUTO-20260606T202647 : Restore Operation Failed with EOF Error on Azure SQL MI Backup Restore

## Classification
- **Category**: RESTORE
- **Severity**: P1
- **Auto-Generated**: Yes
- **Generated On**: 2026-06-06T20:30:00Z

## Problem Description
The restore operation for the database `EnterpriseHR` failed with the error message:
```
EOF when reading a line
```
at 2026-06-06T20:26:47.204331. The restore was initiated from an Azure SQL Managed Instance backup stored as `.bak` file(s) on Azure Blob Storage (container: `backupfromazsqlmi` in the storage account `2276259blob`), restoring to SQL Server 2025 on a Windows VM (IP: 10.10.0.5) via RESTORE FROM URL with SAS credentials. The automation used `restore_database` tool running on a Linux dev VM connected remotely by pyodbc.

## Root Cause Analysis
Based on the environment and failure context:

- **EOF when reading a line** typically means the restore process encountered an unexpected end of file or incomplete file data.

Potential causes within this architecture:

1. **Incomplete or corrupted `.bak` file in Azure Blob Storage**  
   - Upload from SQL MI to Blob may have been interrupted or corrupted, causing truncated backup files.

2. **Incorrect or incomplete Blob storage path or file naming mismatch**  
   - The automation tool auto-discovers `.bak` files by listing the blob container. If naming conventions between `EnterpriseHR` and backup filenames do not match exactly, the wrong or partial file may have been selected.

3. **Expired or invalid SAS token during RESTORE FROM URL**  
   - RESTORE operation relies on SAS credentials to access blob storage. If the SAS token expired or lacked sufficient permissions to read the entire file, partial reads may cause EOF errors.

4. **Network connectivity or firewall issues between SQL Server VM and Blob Storage**  
   - Network interruptions or throttling could cause incomplete streaming of the backup from Blob to SQL Server.

5. **Blob listing failure or partial read by the automation tool on Linux dev VM**  
   - If the tool failed to correctly enumerate all backup files due to SAS expiry or network issues, it might have selected an incomplete backup or an incorrect file.

6. **Multiple backups with similar names causing ambiguous selection**  
   - The tool may have tried to restore a backup that is not a full backup or a differential with missing base backup.

## Resolution Steps

1. **Validate Blob content and backup file integrity**  
   - Use Azure Storage Explorer or Azure CLI (`az storage blob download`) to manually download the `.bak` file corresponding to `EnterpriseHR` backup.  
   - Confirm file size matches source backup expectations (use SQL MI backup history or storage blob properties).  
   - If possible, restore the backup file locally on a test SQL Server to confirm validity.

2. **Check SAS Token validity**  
   - Verify that the SAS token used by the automation tool and SQL Server RESTORE command is current and has at least read permissions to the entire backup blob.  
   - If expired or about to expire, generate a fresh SAS token with appropriate access.  

3. **Verify blob container and path correctness**  
   - Ensure the automated blob listing is targeting container `backupfromazsqlmi` inside storage account `2276259blob`.  
   - Confirm naming conventions align exactly, e.g., backup file might be `EnterpriseHR_FULL_yyyymmdd.bak` or similar.

4. **Check network connectivity and firewall rules**  
   - From the SQL Server VM (10.10.0.5), test connectivity to Azure Blob Storage endpoint using tools like curl or PowerShell to fetch blobs using the same SAS URL.  
   - Review NSG/firewall logs for any blocked connections to Azure Blob storage FQDNs or IPs.

5. **Review automation tool logs**  
   - Check the `restore_database` tool logs for errors during blob list retrieval or file access.  
   - Confirm the tool correctly parses and passes the full blob URL with SAS to the restore command.

6. **Retry restore operation after above validations**  
   - Use verified `.bak` file URL and a fresh SAS token.  
   - Monitor logs for EOF errors.  

7. **If corruption is confirmed, trigger a fresh backup**  
   - Initiate a new full backup from Azure SQL MI directly to Blob Storage and retry restore.

## Prevention

- Implement automated verification of backup file integrity post-upload to Blob Storage (e.g., checksum validation).  
- Monitor SAS token expiration and automate renewal for restore operations.  
- Enforce strict naming conventions for backup files matching database names and timestamps.  
- Harden network configurations and monitor connectivity between SQL Server VM and Blob Storage.  
- Enhance `restore_database` tool to verify backup files via HEAD blob calls or partial reads before restore attempt.

## Suggested Tool Improvement

- Add pre-restore validation step in `restore_database` tool to:  
  - Confirm backup file existence with HEAD request to the Blob URL.  
  - Validate SAS token permissions and expiry prior to restore.  
  - Download partial header metadata to ensure file completeness.  
- Provide clearer error reporting indicating if the EOF error stems from network truncation, SAS permission issues, or corrupted blobs.  
- Support automatic fallback to recent valid backup files based on backup metadata indexed from Blob Storage.  
- Integrate retry logic with refreshed SAS tokens and network connectivity checks before initiating restore.
