# SOP-AUTO-20260606T234449 : Restore Operation Database Name Mismatch Error

## Classification
- **Category**: RESTORE
- **Severity**: P1
- **Auto-Generated**: Yes
- **Generated On**: 2026-06-06T23:44:49.736539Z

## Problem Description
The restore operation for the database `EnterpriseSales` failed with the error:  
"The request is to restore database 'EnterpriseSales' but the previous validation was for database 'saledb'. Please confirm the correct database name to restore before proceeding."

This indicates a mismatch between the expected database name and the database name validated in the backup metadata prior to restore, causing the restore process to halt.

## Root Cause Analysis
- Backups originate on Azure SQL Managed Instance under database name `EnterpriseSales`.
- The automated restore procedure discovers backup `.bak` files by listing the Azure Blob Storage container `backupfromazsqlmi` in Storage Account `2276259blob`.
- This error suggests that the backup file identified by the automation corresponds to a different database (`saledb`) instead of `EnterpriseSales`.  
- Possible causes:
  - The blob container listing returned backup files for multiple databases, and the tool selected a `.bak` file belonging to `saledb` mistakenly instead of `EnterpriseSales`.
  - Backup filenames do not consistently or clearly identify the source database, leading to incorrect selection.
  - SAS token permission or expiry issues may lead to incomplete blob listings, causing an incorrect file to be chosen.
  - Network or firewall constraints between the Linux dev VM (automation host) and Azure Blob Storage may restrict blob container enumeration, resulting in partial/inaccurate results.
  - Naming convention mismatch: the automation may rely on a naming pattern that does not match the actual file names for `EnterpriseSales` backups.
  - Restore script validation logic checks the backup metadata for database name but fails to align it with the requested target DB properly, e.g., cached metadata from a prior validation of `saledb` was used erroneously.
- The root of failure is thus an inconsistent or incorrect backup file selection caused by mismatched database name metadata between the requested restore target and the backup file contents.

## Resolution Steps
1. **Confirm Backup File Correctness**
   - Using Azure Storage Explorer or Azure Portal, list all blob files under container `backupfromazsqlmi` in storage account `2276259blob`.
   - Identify backup files related to the `EnterpriseSales` database by examining naming conventions or by restoring the backup header metadata (using RESTORE HEADERONLY FROM URL command) on a test server if necessary.
2. **Validate SAS Token and Permissions**
   - Verify the SAS token used for `RESTORE FROM URL` has:
     - Read permission on the container and blobs.
     - Is not expired.
   - Renew or regenerate the SAS token if expired or insufficient.
3. **Verify Automation Backup Discovery Logic**
   - Review the automation script running on the Linux dev VM that lists blobs:
     - Confirm that the blob enumeration filters and naming pattern filters correctly isolate `EnterpriseSales` backup files.
     - Add diagnostic logging for the discovered backup file names and associated database names from the backup metadata.
4. **Perform a Manual RESTORE HEADERONLY**
   - Run on the target SQL Server 2025 VM:
     ```sql
     RESTORE HEADERONLY FROM URL = 'https://2276259blob.blob.core.windows.net/backupfromazsqlmi/<backupfile>.bak'
     WITH CREDENTIAL = '<SAS_Credential_Name>';
     ```
   - Confirm the `DatabaseName` field in metadata matches `EnterpriseSales`.
   - Use this to confirm the backup file is the correct one before restore.
5. **Correct Restore Script**
   - Update restore scripts to:
     - Perform backup header validation immediately before restore.
     - Abort and prompt for user intervention if discrepancy in database names is detected.
6. **Retry Restore Operation**
   - Once the correct `.bak` file has been identified and SAS credentials verified, retry the restore process using the validated file URL.
7. **Connectivity Check**
   - If blob listing or access failures persist, verify:
     - Network connectivity and firewall rules between Linux dev VM and Azure Blob Storage.
     - DNS resolution and routing.
     - Access logs on Azure Blob Storage for denied API calls.

## Prevention
- Enforce strict and consistent backup file naming conventions incorporating source database name and timestamp.
- Implement robust blob discovery logic with verification steps to confirm selected backup files match the intended database name before restore.
- Monitor SAS token expiry dates proactively and integrate automated renewal workflows.
- Add automation logging and alerts for backup file discovery anomalies or mismatches.
- Conduct periodic restore rehearsals to validate backup usability and automation logic.
- Document network access requirements and verify connectivity post any infrastructure changes.

## Suggested Tool Improvement
- Enhance automation to perform RESTORE HEADERONLY metadata checks programmatically on candidate backup files prior to any restore attempt.
- Implement database name validation checkpoints with clear error messages and fallback options to select the correct backup file.
- Provide mechanisms to cache and reconcile blob listings with backup metadata to avoid stale or incorrect file usage.
- Integrate SAS token lifecycle management and renewal capability within the automation pipeline.
- Add comprehensive logging for blob enumeration, SAS usage, and restore validation steps to facilitate faster troubleshooting.
