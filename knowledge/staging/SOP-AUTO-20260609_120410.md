# SOP-AUTO-20260609T120300Z : Restore Failure due to Active Connections Blocking Operation

## Classification
- **Category**: RESTORE  
- **Severity**: P1  
- **Auto-Generated**: Yes  
- **Generated On**: 2026-06-09T12:03:07Z  

## Problem Description
The automated database restore operation for the `EnterpriseSales` database on the target SQL Server 2025 VM failed. Despite all pre-flight checks indicating readiness (including verified presence of backup `.bak` files in Azure Blob Storage, sufficient disk space (41.5 GB free), and target database online with no active connections reported), the restore failed due to the presence of 3 active connections on the target database at the time of restore. These active connections originated from application pools and report service connections blocking the restore operation. The automation tooling currently lacks the capability to terminate these connections prior to restore.

## Root Cause Analysis
- The SQL Server restore with `RESTORE FROM URL` requires exclusive access to the target database.
- Although initial checks returned that no active connections were present, concurrent activity by application pools (`AppPool_v2`) and a report service process (`ReportService`) established connections after checks completed and before the restore attempt.
- The automate restore tool does not include functionality to forcibly kill or disconnect active sessions before initiating the restore.
- The network, blob storage access, SAS authorization, and backup file availability were all verified and not implicated.
- The discrepancy between the pre-flight check results and actual restore attempt state indicates a race condition or lag between check completion and restore execution.
- No connectivity, naming, or SAS token issues were identified given the successful backup discovery and verification steps.

## Resolution Steps
1. **Manually Identify and Terminate Active Connections:**
   - Connect to the target SQL Server 2025 instance (10.10.0.5) using an administrative SQL client.
   - Run the following query to identify blocking connections:
     ```sql
     SELECT session_id, login_name, host_name, program_name
     FROM sys.dm_exec_sessions
     WHERE database_id = DB_ID('EnterpriseSales')
       AND is_user_process = 1;
     ```
   - For each session returned, run:
     ```sql
     KILL <session_id>;
     ```
   - Confirm no active connections exist for `EnterpriseSales`.
   
2. **Pause Applications Temporarily:**
   - Notify application teams running `AppPool_v2` and reporting services to pause or stop connections during the restore window.
   - Alternatively configure applications to disconnect during restore from maintenance scripts or CI/CD.

3. **Re-run the Restore Command:**
   - Retry the restore using the original automation tool.
   - Monitor the operation to ensure no new connections are established mid-restore.

4. **Schedule Restore During Maintenance Windows:**
   - Prefer off-hours restore to minimize interference from active connections.

## Prevention
- **Enhance Automation to Handle Active Connections:**
  - Integrate a pre-restore step that checks for and kills active connections.
  - Alternatively, automate a "single-user" mode or ALTER DATABASE restrict access mode to block new connections during restore.
- **Implement Application Connection Throttling:**
  - Coordinate with app teams to reduce connection persistence or enable connection draining during maintenance.
- **Use Maintenance Windows and Alerts:**
  - Define windows during which restores are allowed.
  - Alert DBAs or Ops if active connections detected before restore.

## Suggested Tool Improvement
- Develop or incorporate a tool or script that:
  - Detects active user connections on the target database in real time.
  - Gracefully kills or denies connections before starting restore.
  - Optionally switches the target database to single-user or restricted user mode before restore and reverts after completion.
- Add retry logic with short delay if active connections are detected during restore.
- Add better synchronization between pre-flight validation and restore execution to minimize race conditions.
