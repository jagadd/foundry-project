- **Category**: RESTORE

## Problem Description
A restore operation for the database `EnterpriseHR` on SQL Server 2025 running on a Windows VM (10.10.0.5) failed with the error:

> "The restore attempt was rejected by the operator. To proceed, please confirm if you want me to retry the restore operation or if there are any specific details or a backup path you want me to use for restoring the database 'EnterpriseHR'."

This error occurred during a `RESTORE` command executed remotely via pyodbc automation on a Linux development VM. The restore method uses `RESTORE FROM URL` with a SAS credential pointing to Azure Blob Storage container `backupfromazsqlmi` in storage account `2276259blob`.

## Root Cause Analysis
- **Operator Intervention Required**: The error message indicates the restore operation requires manual confirmation or specific clarification before proceeding. This typically means the automated process detected some ambiguity or policy preventing automatic continuation.
- **Potential Causes:**
  - **Backup File Not Found or Ambiguous**: The automation may have failed to identify a unique valid `.bak` file for `EnterpriseHR` within the `backupfromazsqlmi` blob container. Possible due to:
    - Backup blob naming mismatch compared to the database name.
    - Backup files stored in a subfolder/path not auto-discovered by the tool.
  - **SAS Token Issues**: The SAS credential used might be expired or lack sufficient permissions, resulting in inability to list blobs or verify backup existence.
  - **Network or Firewall Restrictions**: The SQL Server VM or Linux dev VM might be unable to connect to the Azure Blob Storage container for blob enumeration or restore.
  - **Restore Policy or Automation Safeguards**:
    - The `restore_database` tool may have logic to require operator confirmation if the target restore path or backup file is ambiguous.
    - A previous restore attempt may be pending or conflicting, causing the operator to intervene.
  - **Inconsistent Backup Pushes**: The Azure SQL Managed Instance might not have pushed fresh backups to Blob Storage, or backups might be stale or corrupted.
  
No evidence suggests local disk backup files exist, so the restore cannot proceed without valid blobs accessible via URL.

## Resolution Steps
1. **Verify Backup File Existence and Naming:**
   - From the Linux dev VM, run a blob list command against `https://2276259blob.blob.core.windows.net/backupfromazsqlmi` using the current SAS token:
     ```bash
     az storage blob list \
       --account-name 2276259blob \
       --container-name backupfromazsqlmi \
       --sas-token "<SAS_TOKEN>" \
       --output table
     ```
   - Confirm the presence of `.bak` files matching `EnterpriseHR`, or check for any folder/prefix that contains matching backups.
2. **Confirm SAS Token Validity and Permissions:**
   - Check SAS token expiry and permissions (read, list, and get).
   - Renew or generate a new SAS token with appropriate scoped access if expired or insufficient.
3. **Check Network Connectivity:**
   - From the Windows VM and Linux dev VM, test connectivity to Azure Blob Storage endpoint.
     ```bash
     curl -v "https://2276259blob.blob.core.windows.net/backupfromazsqlmi?restype=container&comp=list&<SAS_TOKEN>"
     ```
   - Troubleshoot firewall or NSG restrictions if needed.
4. **Check Automation Logs and Confirm Operator Requirement Logic:**
   - Review the `restore_database` tool logs to identify why confirmation was required.
   - If the tool detected multiple backup files or missing backup paths, explicitly specify the correct blob URL backup path when triggering restore.
5. **Retry Restore with Explicit Restore Source:**
   - Use the exact backup blob URL path in the restore command:
     ```sql
     RESTORE DATABASE [EnterpriseHR]
     FROM URL = 'https://2276259blob.blob.core.windows.net/backupfromazsqlmi/<backup_file>.bak'
     WITH CREDENTIAL = '<SAS_Credential_Name>';
     ```
   - Execute the restore either manually or update automation to pass explicit backup path parameter.
6. **Verify No Conflicting Restore or Database States:**
   - Ensure no existing restore operation is in progress or database is not in a restoring state.
   - Drop or rename any existing `EnterpriseHR` database if required.
7. **Coordinate with Operator if Manual Confirmation is Part of Policy:**
   - If automation is designed to pause for manual approval, confirm decision with DBA team before resuming.

## Prevention
- Implement automated validation to ensure:
  - The backup files for each database are consistently named following a strict naming convention including the database name.
  - Backup pushes from Azure SQL Managed Instance complete successfully and regularly.
  - SAS tokens used by automation are monitored and rotated before expiry.
  - Automation supports specifying explicit backup blob URLs or intelligently resolves unique backup files per database.
- Schedule periodic testing of connectivity and SAS token permissions.
- Document and automate operator confirmation workflows or remove manual pauses if possible.

## Suggested Tool Improvement
- Enhance `restore_database` tool with:
  - Automatic detection of ambiguous or missing backup files, with detailed error messages and suggestions.
  - Automatic retry logic with fresh blob list after SAS renewal.
  - Ability to accept explicit backup blob URL overrides via command parameters.
  - Integration with Azure Key Vault to automatically refresh SAS tokens.
  - Option to disable operator confirmation prompt for fully automated CI/CD environments with proper safeguards.
