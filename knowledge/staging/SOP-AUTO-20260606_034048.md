# SOP-AUTO-2026-06-06T033928 : Restore Failure - Backup File Not Found in Azure Blob Storage

## Classification
- **Category**: RESTORE
- **Severity**: P1
- **Auto-Generated**: Yes
- **Generated On**: 2026-06-06T03:39:28.029839Z

## Problem Description
A RESTORE operation for the database "salesdb" failed because no .bak backup files were found at the expected Azure Blob Storage location.  
The restore process uses RESTORE FROM URL with SAS credentials against the container `backupfromazsqlmi` in storage account `2276259blob`.  
Failure details indicate the backup file URL/path is either missing, incorrect, or inaccessible, causing the restore to fail with the state "Not restored - backup file unavailable."

## Root Cause Analysis
- **Backup file absence or naming mismatch:**  
  The backup automation pushes .bak files from Azure SQL Managed Instance (SQL MI) to Azure Blob Storage. The expected file naming convention must align exactly with the "salesdb" database and the latest COPY_ONLY backup with CHECKSUM and COMPRESSION.  
  If backups were not pushed or the file is named unexpectedly, the restore tool’s blob container listing will not locate the correct file.

- **Incorrect container or blob path:**  
  The restore expects files in container `backupfromazsqlmi` under certain virtual folder(s) or root. If the folder path or blob virtual directory is wrong in the restore URL, the file can be “not found.”

- **SAS token issues:**  
  The SAS credential used to access the blob storage for RESTORE FROM URL might have expired, insufficient permissions, or be malformed, resulting in failed blob listing or access.

- **Network or firewall constraints:**  
  The restore automation runs on a Linux dev VM remotely connecting to SQL Server on a Windows VM. Network restrictions between the SQL Server VM and Azure Blob Storage could block access.

- **Automation tool failure or listing issues:**  
  The automation tool discovers files by listing the blob container. If listing fails (due to SAS expiry, permissions, or API issues), the restore command is given a non-existent backup URL.

- **Mismatch with backup requirements:**  
  The restore requires the most recent COPY_ONLY backup with CHECKSUM and COMPRESSION. If such a backup was not generated or uploaded, restore fails.

## Resolution Steps
1. **Verify Backup Presence on Azure Blob Storage:**  
   - Access Azure Portal or use Azure Storage Explorer to open storage account `2276259blob`.  
   - Navigate to container `backupfromazsqlmi`.  
   - Confirm presence of `.bak` files matching `salesdb` and ensure at least one recent COPY_ONLY backup exists with compression/checksum enabled.

2. **Check Blob Naming and Path:**  
   - Confirm the exact blob path (including any virtual directories) where the backup files reside.  
   - Verify naming conventions match the expected pattern for the restore automation tooling.

3. **Validate SAS Token:**  
   - Ensure the SAS token URL used in RESTORE FROM URL is valid, unexpired, and grants at least Read (r) and List (l) permissions on the container or specific blobs.  
   - Renew or regenerate SAS tokens if expired or insufficient.

4. **Test Blob Access from SQL Server VM:**  
   - From the SQL Server 2025 Windows VM (10.10.0.5), test connectivity to the Azure Blob Storage endpoint using the exact RESTORE URL and SAS token.  
   - Use PowerShell or azcopy to attempt reading/listing blobs to verify network/firewall rules allow outbound HTTPS traffic to Azure Blob Storage endpoints.

5. **Update Automation Configuration:**  
   - Update the restore automation's blob container listing parameters or blob path to align with the verified backup location and naming.  
   - Ensure automation uses the valid SAS token and presents the correct RESTORE FROM URL command.

6. **Perform a Manual Restore Test:**  
   - Manually execute a RESTORE DATABASE command on SQL Server 2025 using the verified RESTORE FROM URL and SAS token referencing the confirmed backup file to validate the end-to-end process.

7. **Document and Communicate Findings:**  
   - Update incident report INC-4821 with backup file location and SAS token expiry information.  
   - Inform DBA and DevOps teams of required renewals or corrections.

## Prevention
- Automate SAS token renewal before expiry and incorporate token validity checks in the restore automation tool.  
- Implement monitoring of backup file uploads to Azure Blob Storage, including validation of naming conventions and file integrity.  
- Schedule regular validation tests of backup discovery and restore commands on a staging environment.  
- Maintain network/firewall configurations to allow uninterrupted access from SQL Server VMs to Azure Blob Storage endpoints.  
- Document and enforce a strict naming convention policy for backup files pushed from Azure SQL MI.

## Suggested Tool Improvement
- Enhance the restore automation tool to:  
  - Perform pre-restore validation to confirm backup file existence via blob listing, SAS token validity, and network accessibility, failing fast with detailed error messages.  
  - Automatically refresh SAS tokens or fetch them securely from Azure Key Vault or managed identities.  
  - Log detailed diagnostics on blob listing failures to quickly isolate cause (permission, network, naming).  
  - Support configurable blob path patterns and fuzzy matching to handle slight deviations in naming conventions.
