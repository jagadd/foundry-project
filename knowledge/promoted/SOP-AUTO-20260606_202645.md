# SOP-AUTO-20260606T202555 : Restore Failure - Enforce RESTORE FROM URL using Azure Blob Storage Backups

## Classification
- **Category**: RESTORE
- **Severity**: P1
- **Auto-Generated**: Yes

## Problem Description
An attempt to restore the EnterpriseHR database to the SQL Server 2025 VM (10.10.0.5) failed because the restore operation tried to use a local .bak backup file instead of performing RESTORE FROM URL using the Azure Blob Storage backup as mandated by SOP-RESTORE-001. The restore operation was rejected due to non-compliance with the SOP’s requirement for COPY_ONLY backups less than 48 hours old, with CHECKSUM and COMPRESSION enabled, restored exclusively via Azure Blob Storage RESTORE FROM URL method using a SAS credential.

## Root Cause Analysis
- **Incorrect Restore Method**: The restore process attempted to restore from a .bak file presumably local or from an unsupported path, violating the SOP-RESTORE-001 which mandates RESTORE FROM URL with Azure Blob Storage backups.
- **Backup Location Assumptions**: The automation or operator may have assumed local disk availability for the backup file. However, backups are exclusively stored as .bak files in the Azure Blob Storage container `backupfromazsqlmi` in storage account `2276259blob`.
- **SAS Credential Usage**: RESTORE FROM URL requires a valid SAS token with appropriate read permissions on the blob storage backup file. Any invalid or expired SAS token would prevent access, although this was not explicitly reported.
- **Backup Compliance**: The restore must use backups that are COPY_ONLY, compressed and checksummed, and less than 48 hours old to ensure consistency and reliability—a direct file restore may have targeted a non-compliant backup.
- **Automation Tool Limitation**: The `restore_database` tool used is not configured or allowed to run restores from local .bak files in this environment, only from URLs on blob storage. Attempting a different method caused the operation to be rejected.
- **Potential Network or Configuration Issues**: Although not explicitly seen, common issues include:
  - Blob container or file not discovered by the automation due to listing failures.
  - SAS token expired or lacks read/list permissions.
  - Networking/firewall rules restricting reachability of Azure Blob Storage from SQL Server VM or Linux automation VM.

## Resolution Steps
1. **Verify Backup Availability and Compliance**
   - Log into Azure Portal or use Azure CLI to list blobs in `backupfromazsqlmi` container in storage account `2276259blob`.
   - Confirm presence of a valid EnterpriseHR `.bak` backup file:
     ```bash
     az storage blob list --account-name 2276259blob --container-name backupfromazsqlmi --query "[?contains(name, 'EnterpriseHR')]" --output table
     ```
   - Confirm backup file is a COPY_ONLY backup taken within the last 48 hours with CHECKSUM and COMPRESSION enabled (via backup metadata or naming conventions).
   
2. **Validate SAS Token**
   - Ensure the SAS token used for RESTORE FROM URL has not expired.
   - Verify it grants at least Read and List permissions on the container and backup blob.
   - Renew SAS token if expired or permissions insufficient.

3. **Construct RESTORE FROM URL Command**
   - Retrieve the full authenticated URL for the backup (.bak) file in blob storage.
   - Example format:
     ```
     https://2276259blob.blob.core.windows.net/backupfromazsqlmi/EnterpriseHR_backup_YYYYMMDDHHMMSS.bak?<sas-token>
     ```
   
4. **Pre-Restore Checks**
   - Ensure target SQL Server VM (10.10.0.5) database `EnterpriseHR` is offline:
     - Disconnect all connections.
     - Set database to SINGLE_USER mode.
   - Verify sufficient disk space on SQL Server VM for data and log files.
   - Map database files correctly by verifying logical file names with `RESTORE FILELISTONLY FROM URL`.

5. **Execute Restore Using RESTORE FROM URL**
   - Run T-SQL on SQL Server 2025 VM exactly as:
     ```sql
     RESTORE DATABASE EnterpriseHR
     FROM URL = 'https://2276259blob.blob.core.windows.net/backupfromazsqlmi/EnterpriseHR_backup_YYYYMMDDHHMMSS.bak?<sas-token>'
     WITH MOVE 'EnterpriseHR_Data' TO 'D:\SQLData\EnterpriseHR.mdf',
          MOVE 'EnterpriseHR_Log' TO 'D:\SQLLogs\EnterpriseHR.ldf',
          NORECOVERY, CHECKSUM, REPLACE;
     ```
   - Adjust MOVE paths according to actual VM SQL Server data directory structure.

6. **Post-Restore Steps**
   - Set database to MULTI_USER mode.
   - Run:
     ```sql
     EXEC sp_change_users_login 'AUTO_FIX', 'your_orphaned_user';
     EXEC sp_updatestats;
     ```
   - Execute any required data masking or verification scripts.
   - Confirm availability and connectivity of the restored database.

7. **Schedule or Trigger Automation**
   - Ensure automation tool uses the RESTORE FROM URL method only.
   - If manual intervention is needed, provide the exact blob URL and SAS token for the restore.
   - Correct any tool or scripts attempting to restore from local .bak files with enforced SOP compliance.

## Prevention
- Enforce SOP-RESTORE-001 compliance by disabling or restricting local disk backup restore attempts on SQL Server 2025 VM.
- Automate regular validation of backups in Azure Blob Storage for freshness, COPY_ONLY type, checksum and compression.
- Monitor SAS tokens and automate renewal before expiry.
- Maintain proper firewall and network access rules to allow SQL Server VM to read from Azure Blob Storage.
- Enhance automation tooling to strictly discover and operate only on valid backups from blob storage URLs.
- Provide DBA and automation teams training on correct restore procedures aligned with this SOP.

## Suggested Tool Improvement
- Enhance `restore_database` tool to:
  - Automatically list and select only valid COPY_ONLY backups from the Azure Blob container.
  - Validate SAS token expiration and renew or alert proactively.
  - Reject any restore requests attempting direct local .bak file restores with explicit feedback.
  - Include pre-flight checks for database state, disk space, and backup file metadata before initiating restore.
  - Integrate logging with clear error messages guiding operators toward SOP-compliant restore procedures.
- Implement URL-based restore templates and parameterization to minimize manual errors.
